# src-tauri/src/

**Coração do app.** Onde mora a lógica de negócio em Rust.

## Módulos

### `main.rs` — entrypoint

Apenas chama `falcao_launcher_lib::run()`. Wrapper porque o crate é `lib + bin` (necessário pro `#[cfg_attr(mobile, ...)]` funcionar). Praticamente nunca muda.

### `lib.rs` — bootstrap do Tauri

Registra:
- Plugins (`tauri_plugin_opener`)
- State global (`ProcessState`)
- Comandos invocáveis do JS (`invoke_handler`)

**Quando adicionar novo comando Rust:**
1. Implementar como `#[tauri::command]` em algum módulo (`process.rs`, `scanner.rs`, `config.rs`, `external.rs`, `icon.rs`, ou criar novo)
2. Adicionar à lista do `tauri::generate_handler![...]` aqui

### `scanner.rs` — descobre projetos em `~/Projects` + extras

Função pública: `scan() -> Vec<Project>`.

Faz:
1. Lista subpastas de `~/Projects` (depth 1, ignora dotfiles)
2. **Concatena com `extra_paths` da config** (paths arbitrários adicionados pelo usuário). Se um path extra colide com um automático, o id ganha sufixo `@<path>`.
3. Pra cada uma: lê `package.json`, extrai scripts, escolhe o melhor (`dev:all` > `dev` > `start:dev` > `start`)
4. Detecta package manager pelo lockfile (`pnpm-lock.yaml` → pnpm, `yarn.lock` → yarn, `bun.lock(b)` → bun, `package-lock.json` → npm, fallback pnpm)
5. **Resolve favicon nessa ordem**: `custom_icon_path` da config (se setado) → `FAVICON_CANDIDATES` (24 caminhos comuns: SVG/PNG genéricos, PWA standards `icon-192/512.png`, `apple-touch-icon.png`, `favicon.ico` por último, `public/assets/images/...`, monorepos, frameworks com `static/`). Encoda como base64 data URI (limite 512 KB).
6. Marca `hidden: bool` se o id está em `config.hidden`. Marca `extra: bool` se veio de `extra_paths`.
7. Retorna ordenado alfabeticamente.

**Comando público adicional**: `list_icon_candidates(project_path) -> Vec<IconCandidate>` — varre 8 pastas comuns (`public/`, `src/assets/`, `static/`, `public/assets/images/`, `apps/web/public/`, etc) por imagens (`.svg/.png/.ico/.webp/.jpg/.jpeg/.gif`) cujo nome contenha `favicon`/`icon`/`logo`/`brand`/`apple-touch-icon`. Retorna `{ relative_path, data_uri, size_bytes }` ordenado pelo menor.

**Constantes que talvez você queira mexer:**
- `PREFERRED_SCRIPTS` — ordem de preferência do script
- `FAVICON_CANDIDATES` — caminhos onde procurar favicon
- `ICON_DIRS` / `ICON_EXTS` — usado por `list_icon_candidates`
- `MAX_FAVICON_BYTES` — 512 KB

### `process.rs` — supervisor de processos (☢️ sensível)

Onde vive a lógica de spawn/kill. **Mexer com cuidado** — bugs aqui deixam processos zumbis.

**Comandos expostos:**
- `start_project(id, path, script, package_manager)` — spawna `<pm> [run] <script>` no path. **Aloca portas livres** se há override no config (frontend_port/backend_port), incrementando ±100 a partir da preferida ou caindo pra `bind(":0")`. Injeta `PORT` e/ou `BACKEND_PORT` no env do `Command`.
- `stop_project(id)` — manda SIGTERM no pgroup, SIGKILL após 5s se não morrer
- `running_ids()` — lista projetos rodando (usado no boot pra reidratar UI)

**State:**
- `ProcessState` é um `Mutex<HashMap<String, ChildHandle>>`
- `ChildHandle` guarda um `oneshot::Sender<()>` — usado pra sinalizar abort sem ter ownership do `Child`

**Eventos emitidos:**
- `log` — uma linha de stdout/stderr
- `status` — `running` / `stopped` / `crashed`, com exit code
- `port` — primeira porta detectada (regex em ambos os streams), só uma vez por processo. **Filtra a porta configurada como `backend_port`** (assim a regex captura a do frontend em projetos full-stack)
- `port-allocated` — `{ id, frontend_port?, backend_port? }` — emitido logo após spawn quando há override de porta. UI exibe a porta esperada antes do regex confirmar.

**Coisas que parecem opcionais mas NÃO são:**
- `process_group(0)` na criação do `Command` — sem isso, matar `pnpm` deixa `vite` orfão.
- `kill(-pgid, ...)` (negativo!) na hora de matar — kernel interpreta isso como "todo o grupo". Sem o sinal negativo, mata só o pnpm.
- `kill_on_drop(true)` — safety net se a task do supervisor for cancelada inesperadamente.
- `tokio::select!` no supervisor — alternativa (`while let` com check de canal) tem race conditions.

**Testes:** módulo `tests` no fim do arquivo cobre o regex de porta. Rodar com `cargo test`. Adicionar teste novo sempre que mexer no regex.

### `ports.rs` — alocação de portas livres

Função pública: `find_free_port(preferred: Option<u16>) -> Option<u16>`.

Tenta `preferred` via `TcpListener::bind`. Se ocupada, incrementa até 100 portas. Fallback `bind(":0")` (kernel-assigned). 3 testes próprios. Usado pelo `process.rs` antes de spawn.

### `config.rs` — persistência por usuário

Schema gravado em `~/.config/falcao-launcher/config.json`:

```json
{
  "version": 1,
  "projects": {
    "<id>": {
      "frontend_port": 5173,
      "backend_port": 8100,
      "custom_icon_path": "public/icon-192x192.png"
    }
  },
  "hidden": ["projeto-x", "outro"],
  "extra_paths": ["/home/falcao/work/projeto-fora-de-projects"]
}
```

**Comandos expostos:**
- `get_project_config(id)` / `set_project_config(id, config)` — override de portas + ícone custom por projeto.
- `set_project_hidden(id, hidden: bool)` — toggle.
- `add_extra_path(path)` — valida que é dir, canonicaliza, adiciona ao set.
- `remove_extra_path(path)`.

**Funções públicas internas** (usadas pelo `scanner.rs` e `process.rs`):
- `project(id) -> ProjectConfig` — defaults vazios.
- `is_hidden(id) -> bool`.
- `extra_paths() -> Vec<String>`.

### `external.rs` — abrir editor / terminal

**Comandos expostos:**
- `open_in_editor(path)` — spawna `code <path>` detached (stdio null, sem supervisão).
- `open_in_terminal(path)` — spawna `ghostty --working-directory=<path>` detached.

Usa `std::process::Command` síncrono (fork+exec é rápido, não precisa async). Validação `path.is_dir()` antes de spawnar.

### `icon.rs` — resolução de ícones do tema do sistema

**Comando exposto:**
- `resolve_icon(name, size) -> Option<String>` (data URI base64).

Usa a crate `freedesktop-icons` pra seguir XDG icon spec. Lê o tema ativo do GNOME via `gsettings get org.gnome.desktop.interface icon-theme` (cacheado em `Lazy<Option<String>>`). Se gsettings falhar, fallback pra `lookup` sem theme. Limite 512 KB.

Frontend usa via `<SystemIcon name="visual-studio-code" fallback={...} />` que tem cache de Map estático e fallback pra PNGs hardcoded em `src/assets/`.

## Convenções

- **Comandos públicos** (`#[tauri::command]`) ficam no módulo onde a lógica vive, não centralizados.
- **Tipos serializados pro frontend** ficam `pub` e em `serde::Serialize` derive — manter os campos em sync com `src/types.ts`.
- **Erros voltam como `Result<T, String>`** — Tauri serializa o `Err(String)` direto pro reject da promise no JS. Não usar `anyhow::Error` aqui (não serializa).
- **Async em comando** = adicionar `async` na assinatura. Tauri usa `tauri::async_runtime` (que é tokio). Pra spawnar tasks dentro de comandos: `tauri::async_runtime::spawn(async move { ... })`.
- **Manter funções <60 linhas.** Se ficou maior, fatorar.
