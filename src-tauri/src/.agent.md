# src-tauri/src/

**Coração do app.** Onde mora a lógica de negócio em Rust.

## Módulos

### `main.rs` — entrypoint

Apenas chama `falcao_launcher_lib::run()`. Wrapper porque o crate é `lib + bin` (necessário pro `#[cfg_attr(mobile, ...)]` funcionar). Praticamente nunca muda.

### `lib.rs` — bootstrap do Tauri

Registra:
- Plugins (`tauri_plugin_opener`)
- State global (`ProcessState`)
- Comandos invocáveis do JS (`invoke_handler`)

**Quando adicionar novo comando Rust:**
1. Implementar como `#[tauri::command]` em algum módulo (`process.rs`, `scanner.rs`, ou criar novo)
2. Adicionar à lista do `tauri::generate_handler![...]` aqui

### `scanner.rs` — descobre projetos em `~/Projects`

Função pública: `scan() -> Vec<Project>`.

Faz:
1. Lista subpastas de `~/Projects` (depth 1, ignora dotfiles)
2. Pra cada uma: lê `package.json`, extrai scripts, escolhe o melhor (`dev:all` > `dev` > `start:dev` > `start`)
3. Detecta package manager pelo lockfile (`pnpm-lock.yaml` → pnpm, `yarn.lock` → yarn, etc)
4. Procura favicon em 17 caminhos comuns (`public/favicon.svg`, `apps/web/public/favicon.png`, etc), encoda como base64 data URI (limite 256 KB)
5. Retorna ordenado alfabeticamente

**Constantes que talvez você queira mexer:**
- `PREFERRED_SCRIPTS` — ordem de preferência do script
- `FAVICON_CANDIDATES` — caminhos onde procurar favicon
- `MAX_FAVICON_BYTES` — limite de tamanho

### `process.rs` — supervisor de processos (☢️ sensível)

Onde vive a lógica de spawn/kill. **Mexer com cuidado** — bugs aqui deixam processos zumbis.

**Comandos expostos:**
- `start_project(id, path, script, package_manager)` — spawna `<pm> [run] <script>` no path
- `stop_project(id)` — manda SIGTERM no pgroup, SIGKILL após 5s se não morrer
- `running_ids()` — lista projetos rodando (usado no boot pra reidratar UI)

**State:**
- `ProcessState` é um `Mutex<HashMap<String, ChildHandle>>`
- `ChildHandle` guarda um `oneshot::Sender<()>` — usado pra sinalizar abort sem ter ownership do `Child`

**Eventos emitidos:**
- `log` — uma linha de stdout/stderr
- `status` — `running` / `stopped` / `crashed`, com exit code
- `port` — primeira porta detectada (regex em ambos os streams), só uma vez por processo

**Coisas que parecem opcionais mas NÃO são:**
- `process_group(0)` na criação do `Command` — sem isso, matar `pnpm` deixa `vite` orfão.
- `kill(-pgid, ...)` (negativo!) na hora de matar — kernel interpreta isso como "todo o grupo". Sem o sinal negativo, mata só o pnpm.
- `kill_on_drop(true)` — safety net se a task do supervisor for cancelada inesperadamente.
- `tokio::select!` no supervisor — alternativa (`while let` com check de canal) tem race conditions.

**Testes:** módulo `tests` no fim do arquivo cobre o regex de porta. Rodar com `cargo test`. Adicionar teste novo sempre que mexer no regex.

## Convenções

- **Comandos públicos** (`#[tauri::command]`) ficam no módulo onde a lógica vive, não centralizados.
- **Tipos serializados pro frontend** ficam `pub` e em `serde::Serialize` derive — manter os campos em sync com `src/types.ts`.
- **Erros voltam como `Result<T, String>`** — Tauri serializa o `Err(String)` direto pro reject da promise no JS. Não usar `anyhow::Error` aqui (não serializa).
- **Async em comando** = adicionar `async` na assinatura. Tauri usa `tauri::async_runtime` (que é tokio). Pra spawnar tasks dentro de comandos: `tauri::async_runtime::spawn(async move { ... })`.
- **Manter funções <60 linhas.** Se ficou maior, fatorar.
