# monitor — agent.md

## Propósito
Lado launcher do monitor: SSH tunnel pro Postgres da VM + queries SELECT (somente leitura) + commands Tauri expostos pro frontend. Não escreve no DB — quem escreve é o `monitor-agent` rodando na VM.

## Arquivos
- `mod.rs` — re-exports do módulo.
- `tunnel.rs` — `TunnelManager` que gerencia o processo `ssh -L` (porta local 54322 → 5432 na VM). Singleton mantido em `MonitorState` via `tauri::manage`.
- `queries.rs` — funções SELECT (`vm_status`, `list_containers`, `fetch_metric_series`, `fetch_health_summary`). Validação de inputs pra prevenir injection. Tipos públicos (`ContainerInfo`, `HealthCheckSummary`) consumidos pelo `stacks.rs`.
- `stacks.rs` — queries pras "Stacks em produção" (Sprint 2): `list_stacks` (DISTINCT do JOIN entre `metrics.labels.stack` e `vercel_deployments.project_name`) e `stack_detail` (último deploy + histórico 10 deploys + containers + endpoint health match best-effort).
- `commands.rs` — `#[tauri::command]` invocáveis do frontend: tunnel (`monitor_open_tunnel`, `monitor_close_tunnel`), VM (`monitor_vm_status`, `monitor_list_containers`, `monitor_metric_series`, `monitor_fetch_logs`), health (`monitor_health_summary`), stacks (`monitor_list_stacks`, `monitor_stack_detail`).

## Padrões
- Tunnel é singleton mantido em `MonitorState` (Tauri `manage`). Abrir o segundo tunnel reutiliza o existente.
- Pool Postgres (`deadpool-postgres`) só existe enquanto o tunnel está aberto. Fechar o tunnel dropa o pool.
- Logs do container são fetchados on-demand via SSH (`docker logs --tail`) — NÃO persistidos em DB.
- Inputs do frontend (nome de container, source) são validados em `queries.rs` antes de entrar em qualquer SQL/SSH command.

## Decisões recentes
- 2026-05-06: implementado na Fase 1 do monitor.
- Tunnel via `ssh -L` (processo nativo) em vez de `russh` puro Rust — mais simples, mesma segurança, e Falcão já tem chaves SSH configuradas.
- Porta local 54322 fixa (não conflita com Postgres local que usa 5432).
- 2026-05-06: `MonitorState::new()` agora lê `MONITOR_READER_PASSWORD` de `~/.config/falcao-launcher/.env` como fallback quando a env var está vazia (caso clássico: app aberto via atalho do GNOME, que não herda env do shell). Env var ainda tem precedência. Parser simples (linhas `KEY=VALUE`, aceita aspas, ignora `#` e linhas vazias) — ver `read_password_from_config()` em `commands.rs`. Arquivo é manual (chmod 600), nunca vive no repo.
- 2026-05-07 (Sprint 2 — Vercel stacks): `stacks.rs` novo. Cruza duas dimensões em query-time: containers com label `monitor.stack` (vindos de `metrics.labels`) + projetos Vercel (vindos de `vercel_deployments`). Match por nome igual. `StackDetail.vercel_history` traz até 10 deploys via DISTINCT ON deployment_id (ordenados por created_at desc). Endpoint health match é best-effort via ILIKE `%name%`.
- 2026-05-07 (Sprint 3 — modo análise): comando novo `monitor_fetch_logs_range` (`docker logs --since/--until` via SSH). Validações: regex container (mesma do `monitor_fetch_logs`), range max 24h, until > since. Retorna `LogsRangeResponse { text, truncated, line_count }`. 7 testes unitários nos helpers (parse RFC3339, range math, regex). Constantes: `MAX_RANGE_HOURS = 24`, `MAX_TAIL_LINES = 2000`.
- 2026-05-08 (Sprint B1 — Snyk-like): `security.rs` novo. Queries `list_vulnerabilities(severities, kinds)` (DISTINCT ON dedup + whitelist defensiva + filter state=open + ts>7d), `vuln_summary()` (contadores por severidade), `vuln_count_by_repo()` (kind=deps Critical/High pra chips dos ProjectCards). 3 comandos Tauri novos em `commands.rs`. Tipos `VulnerabilityRow`, `VulnSummary` serializáveis pro frontend.

## Pendências conhecidas
- Tunnel não tem auto-reconnect; se SSH cair, o frontend precisa pedir reabrir.
