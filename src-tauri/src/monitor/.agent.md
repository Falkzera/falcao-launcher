# monitor — agent.md

## Propósito
Lado launcher do monitor: SSH tunnel pro Postgres da VM + queries SELECT (somente leitura) + commands Tauri expostos pro frontend. Não escreve no DB — quem escreve é o `monitor-agent` rodando na VM.

## Arquivos
- `mod.rs` — re-exports do módulo.
- `tunnel.rs` — `TunnelManager` que gerencia o processo `ssh -L` (porta local 54322 → 5432 na VM). Singleton mantido em `MonitorState` via `tauri::manage`.
- `queries.rs` — funções SELECT (`vm_status`, `list_containers`, `fetch_metric_series`). Validação de inputs pra prevenir injection.
- `commands.rs` — `#[tauri::command]` invocáveis do frontend (`monitor_open_tunnel`, `monitor_close_tunnel`, `monitor_vm_status`, `monitor_list_containers`, `monitor_metric_series`, `monitor_fetch_logs`).

## Padrões
- Tunnel é singleton mantido em `MonitorState` (Tauri `manage`). Abrir o segundo tunnel reutiliza o existente.
- Pool Postgres (`deadpool-postgres`) só existe enquanto o tunnel está aberto. Fechar o tunnel dropa o pool.
- Logs do container são fetchados on-demand via SSH (`docker logs --tail`) — NÃO persistidos em DB.
- Inputs do frontend (nome de container, source) são validados em `queries.rs` antes de entrar em qualquer SQL/SSH command.

## Decisões recentes
- 2026-05-06: implementado na Fase 1 do monitor.
- Tunnel via `ssh -L` (processo nativo) em vez de `russh` puro Rust — mais simples, mesma segurança, e Falcão já tem chaves SSH configuradas.
- Porta local 54322 fixa (não conflita com Postgres local que usa 5432).

## Pendências conhecidas
- Tunnel não tem auto-reconnect; se SSH cair, o frontend precisa pedir reabrir.
