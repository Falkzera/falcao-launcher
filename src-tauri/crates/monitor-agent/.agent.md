# monitor-agent — agent.md

## Propósito
Binário standalone que roda como systemd user service na VM Hetzner (`falcao-main`, IP `162.55.217.189`). Coleta métricas a cada 15s (VM-level via `/proc`, containers Docker via `docker stats`, Hetzner via `hcloud`) e escreve no Postgres+TimescaleDB local em `/opt/falcao-monitor/`.

## Arquivos
- `Cargo.toml` — manifesto (bin).
- `src/main.rs` — bootstrap (env, logging) + main loop (poll a cada 15s, heartbeat, buffer fallback).
- `src/db.rs` — pool `deadpool-postgres` + writer (`insert_batch`, `write_heartbeat`).
- `src/buffer.rs` — buffer in-memory pra resiliência quando DB cai (até ~50k linhas / ~30 min).
- `src/collectors/` — coletores (vm, container, hetzner). Ver `.agent.md` da pasta.

## Padrões
- Um coletor = um arquivo em `collectors/`. Função `pub async fn collect(ts: DateTime<Utc>) -> Result<Vec<MetricRow>>`.
- Erros não-fatais (coletor individual falhou, hcloud rate-limited, etc.) logados como `warn` e o loop continua. Só erros catastróficos abortam.
- Quando DB cai, o buffer guarda linhas em RAM e re-tenta no próximo tick. Drop oldest quando cheio.
- Deploy: `./scripts/deploy-monitor-agent.sh` (build release local + scp + restart systemd).

## Decisões recentes
- 2026-05-06: criado pra Fase 1 do monitor. Spec `docs/superpowers/specs/2026-05-06-vm-monitor-fase-1-design.md`.
- Tempo de poll fixo em 15s (`POLL_INTERVAL_SECS` em `monitor-shared`).
- Heartbeat numa tabela dedicada (`agent_heartbeat`) — frontend usa pra detectar agente caído.
- 2026-05-07 (v0.2.0, Sprint 2 — Vercel stacks): adicionado coletor Vercel em task paralela (poll 5min via `tokio::time::interval`, separada do loop de 15s). Lê `VERCEL_TOKEN` do env via systemd `EnvironmentFile=-/home/falcao/.config/falcao-monitor/.env`. Sem token → coletor não spawna (graceful). `db.rs` ganhou `insert_vercel_deployments` (transactional batch). `container.rs` agora também lê labels via `docker inspect` e propaga `monitor.stack` em `MetricRow.labels` (JSON `{"stack": "..."}`).

## Pendências conhecidas
- Buffer não é persistido em disco; reboot da VM perde métricas in-flight (aceito pra Fase 1).
- Coletor Vercel não tem retry exponencial em 401/429 — só log warn no próximo tick (5min). Reavaliar se rate limit virar problema com vários projetos.
