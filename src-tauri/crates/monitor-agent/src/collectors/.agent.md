# collectors — agent.md

## Propósito
Coletores de métricas do `monitor-agent`. Cada arquivo é uma fonte distinta de dados.

## Arquivos
- `mod.rs` — re-exports.
- `vm.rs` — métricas VM-level via `/proc/meminfo`, `/proc/loadavg`, `df`, `uptime`. Inclui CPU%, swap, disk usage, load 1/5/15, mem usado/total.
- `container.rs` — métricas por container via `docker stats --no-stream --format json` + `docker inspect` pra ler labels (`monitor.stack` em particular). Inclui parser de tamanhos (`"1.2GiB"` → bytes), net rx/tx, mem limit, status de health, e propagação de `stack` em `MetricRow.labels`.
- `hetzner.rs` — métricas via `hcloud server describe falcao-main -o json` (traffic outbound, status do server, custo acumulado).
- `vercel.rs` — deploys da Vercel via REST API (`GET /v9/projects` + `GET /v6/deployments` com Bearer token). Lista todos os projetos da conta automaticamente (zero allowlist no agente).

## Padrões
- Cada coletor expõe `pub async fn collect(ts: DateTime<Utc>) -> Result<Vec<MetricRow>>`.
- Erro num coletor NÃO derruba o loop principal — `main.rs` cuida disso, loga `warn` e segue.
- Tipos `MetricRow`, `MetricSource` vêm de `monitor-shared`.
- Coletores rodam concorrentemente via `tokio::join!` no main loop, então não devem compartilhar estado mutável.

## Decisões recentes
- 2026-05-06: criados na Fase 1 do monitor.
- 2026-05-06 (review): adicionadas métricas faltantes (cpu_pct, swap, disk, net, mem_limit, health) e fix de URL redaction nos logs.
- 2026-05-07 (Sprint 2 — Vercel stacks):
  - `vercel.rs` novo. Diferente dos outros: NÃO usa CLI externo, fala com api.vercel.com via `reqwest` (Bearer token). Roda em task paralela (poll 5min) — não compartilha tick com os outros (15s).
  - `container.rs` ganhou batch `docker inspect` pra extrair labels. `monitor.stack=<nome>` é propagado pro campo `labels` do `MetricRow` (JSON `{"stack": "..."}`). Container sem a label: `labels: None` (sem regressão pros consumers).
  - Convenção: nome do Vercel project == valor da label `monitor.stack` agrupa as duas dimensões na UI. Override futuro via `monitor.vercel_project=<slug>` quando os nomes divergirem.
- 2026-05-08 (Sprint B3): coletores `vercel_usage.rs` (GET /v1/usage, tick 1h, 4 métricas: bandwidth/build/image_opt/fn) e `gh_actions.rs` (GET /users/{user}/settings/billing/actions, tick 1h, 1 métrica: minutes_used). Ambos com fixture-based parser tests. `hetzner.rs` ganhou função `external_metrics()` que espelha cost_accumulated_usd em `ExternalMetric` pra alimentar a aba Custos. Coletores degradam gracefully sem token (warn + skip task).

## Pendências conhecidas
- `container.rs` depende de Docker Engine local — se o agente for rodar fora da VM, precisa adaptar.
- `vercel.rs` não persiste rate-limit cooldown entre ticks. Se hit repetido, vai logar warn em todos os ticks (mas não vai deteriorar a API).
