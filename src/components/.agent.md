# src/components/

Componentes React específicos do launcher. **Sem biblioteca de UI** (não usamos shadcn-CLI nem nada) — cada componente é escrito à mão, simples, ~30-200 linhas.

## Arquivos

- **`ProjectCard.tsx`** — card de um projeto no grid. Contém:
  - Avatar (favicon real do projeto OU iniciais sobre cor gerada por hash do nome)
  - Nome, path, indicador de status (idle/running/crashed) com pulse animado
  - Chip do package manager + script detectado (`pnpm run dev`, `yarn dev`, etc)
  - Chip da porta (clicável → reabre browser) quando detectada
  - Botão Run/Stop (laranja → vermelho quando rodando)
  - Hander de click na área externa do botão → abre o drawer (callback `onSelect`)
  - Click no botão tem `e.stopPropagation()` pra não disparar `onSelect`

- **`LogsDrawer.tsx`** — drawer lateral direito (480px) com logs ao vivo:
  - Header: bolinha de status, nome do projeto, chip da porta (se houver), botões `clear` e `✕`
  - Body: lista de linhas com cor diferente pra stderr (vermelho suave)
  - Auto-scroll com sticky bottom: usa `stickyRef` em ref pra desligar o scroll automático se o usuário rolar pra cima manualmente
  - Footer: contador de linhas

## Convenções

- **Props sempre tipadas com `type Props = { ... }`**, não `interface`. Mais fácil de estender com `&`.
- **`clsx` pra composição de classes condicionais.** Não usar `cn()` (não temos shadcn).
- **Não exportar default** (exceto componentes que viram lazy-loadable). Usar named exports.
- **Stop propagation em handlers** quando o componente é clicável e tem buttons internos.
- **Não escrever lógica de negócio aqui.** Card e drawer só renderizam props e disparam callbacks. Estado e `invoke()` ficam no `App.tsx` ou são chamados direto via `invoke` (botão Run/Stop são exceção — chamam `invoke` direto porque não impacta state externo).
